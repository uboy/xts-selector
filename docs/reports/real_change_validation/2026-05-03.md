## Graph resolver results

**All 5 test PRs timed out at 600s with `--use-graph-resolver`.**

Root cause: `build_inverted_index()` walks the entire `test/xts/acts/arkui/` tree
(~thousands of .ets files) on every invocation. Without persistent caching of
SDK/ACE/ETS/inverted indices, the graph resolver cannot complete within timeout.

Single-file test (`--changed-file button_model_static.cpp --use-graph-resolver`)
also timed out at 600s, confirming the bottleneck is index construction, not
PR-specific processing.

## Key findings

### F1: AAE population rate improved from 1.6% to 16.26%
The 10x improvement comes from existing signal_inference.py changes that were
deployed in earlier phases. However, 16% is still far below the 50% Phase 7
target. The graph resolver, when cached, is expected to push this to >=50%.

### F2: Required/Optional ratio is 94.9:1 (target <= 8:1)
Most PRs have 0 required targets, with 37-95 optional. This means the ranking
system is not producing strong "must-run" signals. The graph resolver with
inverted index would produce concrete `affected_apis` → `consumer_projects`
mappings that could populate required targets.

### F3: Graph resolver unusable without index cache
This is the **critical blocker** for Phase 7 validation. The graph pipeline
needs:
1. **T9.1 Persistent cache** for SDK index (14K entries), ACE index, and
   inverted index. Estimated 3-5s cold build per index, but 5+ minutes total
   when walking the full ETS tree.
2. Cold build of inverted index on the full `test/xts/acts/arkui/` is the
   main bottleneck — it parses every .ets file with tree-sitter-typescript.
3. Warm cache should bring total graph resolver overhead to <10s per PR.

### F4: Timeout rate at 26% (improved from 53%)
The baseline timeout rate dropped from the documented 53% to 26%, likely due
to the 10-worker parallel approach and better GitCode API handling.

## Remaining gaps for Phase 9

- [ ] **T9.1 Persistent cache** — CRITICAL. Without it, graph resolver is unusable.
      Build indices once, cache to JSON, invalidate on source mtime change.
- [ ] **T9.2 Bridge/Consumer split** — ETS indexer treats all files as test files.
      Need to separate bridge sources (arkoala generated) from consumer test files.
- [ ] **T9.3 Selection reasons** — Per-test "why" in JSON output.
- [ ] **T9.5 Hunk-level resolution** — Filter mappings by changed line ranges.

## Conclusion

Phase 7 production wiring is **structurally complete** but **blocked by performance**.
The graph pipeline correctly resolves changed files to API entities and consumer
tests (verified by 14 unit tests + 5 e2e tests on fixtures), but cannot run in
production without persistent index caching.

**Recommended next step**: Implement T9.1 (persistent cache) and re-run validation
to measure actual AAE rate improvement with graph resolver enabled.
